# SQL Design Reference

Use this reference for relational schema design, schema review, indexing, and migrations.

## Normalization

Normalize to 3NF first for transactional schemas, then denormalize only for a named read path.

| Form | Rule | Violation Example |
|------|------|-------------------|
| 1NF | Atomic values, no repeating groups | `product_ids = '1,2,3'` |
| 2NF | 1NF + no partial dependencies | `customer_name` on `order_items` keyed by `(order_id, product_id)` |
| 3NF | 2NF + no transitive dependencies | `country` derived from `postal_code` on `customers` |

Good 1NF replacement for repeated values:

```sql
CREATE TABLE orders (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  customer_id BIGINT NOT NULL
);

CREATE TABLE order_items (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  order_id BIGINT NOT NULL REFERENCES orders(id),
  product_id BIGINT NOT NULL
);
```

Denormalize when the read path is hot and the duplicate value has a clear owner:

| Scenario | Denormalization Strategy |
|----------|--------------------------|
| Read-heavy reporting | Pre-calculated aggregates |
| Expensive joins | Cached derived columns |
| Analytics dashboards | Materialized views |

## Data Types

Choose types that communicate intent and fit the dialect.

| Need | Preferred Type |
|------|----------------|
| Email or bounded label | `VARCHAR(n)` or dialect text with validation |
| Long body content | `TEXT` |
| Money | `DECIMAL(p,s)` / `NUMERIC(p,s)` |
| Counts and IDs | `BIGINT` when growth is plausible |
| UTC instant in PostgreSQL | `TIMESTAMPTZ` |
| Boolean in PostgreSQL | `BOOLEAN` |

Avoid `VARCHAR(255)` everywhere. Size fields when the domain has a real bound; otherwise use the dialect's idiomatic text type plus validation.

## Constraints

Use constraints to preserve invariants at the database boundary.

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
email TEXT UNIQUE NOT NULL
price NUMERIC(10, 2) NOT NULL CHECK (price >= 0)
```

Foreign-key delete behavior must be intentional:

| Strategy | Use When |
|----------|----------|
| `CASCADE` | Child rows have no meaning without the parent, such as `order_items` |
| `RESTRICT` | Parent deletion should be prevented while important references exist |
| `SET NULL` | Relationship is optional and history should remain |

Polymorphic associations are safer with separate nullable FKs plus a `CHECK` constraint than with `(type, id)` pairs, unless flexibility matters more than referential integrity.

## Relationship Patterns

One-to-many:

```sql
CREATE TABLE orders (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  customer_id BIGINT NOT NULL REFERENCES customers(id)
);
```

Many-to-many:

```sql
CREATE TABLE enrollments (
  student_id BIGINT NOT NULL REFERENCES students(id) ON DELETE CASCADE,
  course_id BIGINT NOT NULL REFERENCES courses(id) ON DELETE CASCADE,
  enrolled_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (student_id, course_id)
);
```

Self-reference:

```sql
CREATE TABLE employees (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  manager_id BIGINT REFERENCES employees(id)
);
```

## Indexing

List query patterns before choosing indexes.

Always consider indexes for:

- Foreign keys.
- Columns used in frequent `WHERE` filters.
- Columns used in frequent `ORDER BY` clauses.
- Unique lookups.

Composite index order follows query shape:

```sql
CREATE INDEX idx_orders_customer_status ON orders(customer_id, status);
```

This helps queries filtering by `customer_id`, or by `customer_id` plus `status`; it does not usually help `WHERE status = 'pending'` alone.

Common pitfalls:

| Pitfall | Problem | Fix |
|---------|---------|-----|
| Over-indexing | Slower writes and larger storage | Keep only indexes with named queries |
| Wrong column order | Planner cannot use index well | Match the leftmost prefix to real filters |
| Missing FK indexes | Slow joins/deletes | Index FK columns explicitly unless verified automatic |

## Migrations

Migrations must account for running application versions and table size.

Adding a required column safely:

```sql
-- 1. Add nullable column.
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- 2. Deploy code that writes the new column.

-- 3. Backfill in bounded batches outside the DDL deploy.
UPDATE users
SET phone = ''
WHERE phone IS NULL
  AND id BETWEEN 1 AND 10000;

-- 4. Enforce after validation.
ALTER TABLE users
  ADD CONSTRAINT users_phone_not_null CHECK (phone IS NOT NULL) NOT VALID;
ALTER TABLE users VALIDATE CONSTRAINT users_phone_not_null;
ALTER TABLE users ALTER COLUMN phone SET NOT NULL;
ALTER TABLE users DROP CONSTRAINT users_phone_not_null;
```

Renaming safely usually means add new column, dual-write, backfill, switch reads, stop old writes, then drop the old column in a later deploy.

For large PostgreSQL tables, create indexes outside a transaction with `CREATE INDEX CONCURRENTLY`.

## Performance Review

Use `EXPLAIN` or the dialect equivalent for slow queries.

Watch for:

- Full table scans on hot paths.
- N+1 query patterns.
- `SELECT *` in high-volume reads.
- Missing pagination on large result sets.
- Aggregations that should become materialized views or cached summaries.
